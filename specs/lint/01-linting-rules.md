# Linting Rules for AI Code Generation

**Last Updated:** January 12, 2026
**Configuration Source:** Project ESLint configuration
**CI Policy:** Zero-tolerance (`--max-warnings 0`)

---

## Overview

All code generated by AI agents (Copilot, Cursor, Claude, etc.) must comply with the complete ESLint configuration. These rules are enforced in the CI/CD pipeline and must pass locally before commit.

---

## ESLint Base Rules Checklist

All ESLint recommended rules apply, plus the custom rules below:

- ☐ `array-callback-return` — Require `return` statements in array method callbacks (severity: error)
- ☐ `constructor-super` — Require `super()` to be called in derived class constructors (error)
- ☐ `for-direction` — Ensure `for` loop update moves the counter in the correct direction (error)
- ☐ `getter-return` — Require `return` statements in getter functions (error)
- ☐ `no-async-promise-executor` — Disallow async functions as Promise executors (error)
- ☐ `no-await-in-loop` — Disallow `await` inside loop bodies (error)
- ☐ `no-compare-neg-zero` — Disallow comparisons against `-0` (error)
- ☐ `no-cond-assign` — Disallow assignment operators in conditional expressions (error)
- ☐ `no-const-assign` — Disallow reassigning `const` variables (error)
- ☐ `no-constant-binary-expression` — Disallow binary expressions where the operation does not affect the value (error)

- ☐ **`semi`** (error) — All statements must end with semicolon
- ☐ **`no-param-reassign`** (error) — Never reassign function parameters

## eslint-plugin-import Rules Checklist

- **`import/order` (error):**
    - ☐ Enforce group order:
        - ☐ Builtin modules (node:fs, node:path, etc.)
        - ☐ External packages (npm, installed packages)
        - ☐ Internal modules (src/ imports)
        - ☐ Parent directories (../, ../../)
        - ☐ Sibling modules (./, ./file)
        - ☐ Index files (./index.js)
        - ☐ Type imports (import type { X } from '...')
    - ☐ Blank line between each group (except before type imports)
    - ☐ Alphabetical sorting within groups (case-insensitive)
    - ☐ Path aliases configured in tsconfig are in the "internal" group
    - ☐ Type imports appear last (separated blank line if mixed)

- **`import/extensions` (error):**
    - ☐ `.js` extension REQUIRED in ESM imports (e.g., `from './file.js'`)
    - ☐ `.ts` extension NOT REQUIRED in TypeScript imports
    - ☐ Ignored packages (node_modules) do not need extensions
    - ☐ ESM modules must have `.js` even if source is `.ts`

- **`import/no-internal-modules` (error):**
    - ☐ Never import from nested paths in packages (no `package/subpath`)
    - ☐ Always import from barrel exports (index.ts) when available

- **`import/no-unused-modules` (error):**
    - ☐ Remove unused files from the codebase
    - ☐ Remove unused exports from modules
    - ☐ Remove unused imports from files
    - ☐ Exception: allow unused exports in entry points (index.ts, main exports)

- **`import/no-cycle` (error):**
    - ☐ No circular dependencies between modules
    - ☐ Verify full import chains (A → B → C → A) and break cycles

- **`import/no-relative-packages` (error):**
    - ☐ Never use relative paths for packages (use package specifier)

- **`import/no-restricted-paths` (error):**
    - ☐ Enforce maximum 3 levels of nested imports
    - ☐ Enforce dependency boundaries between project areas

## eslint-plugin-sonarjs Rules Checklist

### Bug Detection Rules (error)

- ☐ **`sonarjs/no-all-duplicated-branches`** — All branches of if/else or switch do identical thing (likely bug)
- ☐ **`sonarjs/no-element-overwrite`** — Collection element assigned multiple times (likely unintended)
- ☐ **`sonarjs/no-empty-collection`** — Collection created but never used (dead code)
- ☐ **`sonarjs/no-identical-conditions`** — Conditions checked multiple times with identical logic
- ☐ **`sonarjs/no-identical-expressions`** — Same expression appears multiple times (copy-paste bug)

### Code Smell Rules (error)

- ☐ **`sonarjs/cognitive-complexity`** (threshold: 15) — Function logic too complex. Refactor if > 15 complexity score
- ☐ **`sonarjs/max-switch-cases`** (max: 30) — Switch statement has more than 30 cases. Refactor into separate function or pattern
- ☐ **`sonarjs/no-collapsible-if`** — Nested if statements can be combined into single condition
- ☐ **`sonarjs/no-duplicated-string`** (min: 3) — Same string literal appears 3+ times. Extract to constant
- ☐ **`sonarjs/no-gratuitous-expressions`** — Expression always true/false or will always throw. Remove or fix logic
- ☐ **`sonarjs/no-identical-functions`** — Function body duplicated in multiple functions. Extract common logic
- ☐ **`sonarjs/no-inverted-boolean-check`** — Boolean comparison can be simplified (e.g., `!a === true` → `!a`)
- ☐ **`sonarjs/no-nested-switch`** — Switch statement nested inside another switch. Extract nested logic
- ☐ **`sonarjs/no-nested-template-literals`** — Template literals nested (hard to read). Extract variables
- ☐ **`sonarjs/no-one-iteration-loop`** — Loop always executes exactly once. Convert to if statement
- ☐ **`sonarjs/no-redundant-boolean`** — Boolean condition redundant (e.g., `a && true` → `a`)
- ☐ **`sonarjs/no-redundant-jump`** — Jump statement (`return`, `break`) unreachable or unnecessary
- ☐ **`sonarjs/no-same-line-conditional`** — Statement on same line as if/else (readability issue)
- ☐ **`sonarjs/no-small-switch`** — Switch has only 1-2 cases. Use if statement instead
- ☐ **`sonarjs/prefer-object-literal`** — Object built by assignment can use object literal shorthand
- ☐ **`sonarjs/prefer-single-boolean-return`** — Function returns different boolean values. Use single return with expression
- ☐ **`sonarjs/no-unused-collection`** — Collection created, populated, but never used
- ☐ **`sonarjs/no-use-of-empty-return-value`** — Return value of function that returns nothing is used

### Security Rules (error)

- ☐ **`sonarjs/no-eval`** — Never use `eval()`. Major security and performance risk

## Pre-Push Checklist

Before pushing code or requesting review:

- ☐ Lint command passes with no errors or warnings
- ☐ Format check command passes (code is formatted)
- ☐ CI lint command would pass (simulates CI environment locally)
- ☐ No `console.log()` statements left in code
- ☐ No circular imports
- ☐ No unused imports or variables

- No exceptions or eslint-disable comments without justification
